### 1.3 验证结果

- MultiQC 成功读取了 9 个 FASTQC 报告、3 个 flagstat、3 个 idxstats、2 个 stats、1 个 MarkDuplicates 报告
- 报告大小 2.5MB，内容丰富
- HTML 报告包含交互式图表（因 Apple Silicon 架构限制，静态 PNG/PDF/SVG 无法导出）

### 1.4 架构兼容性问题

**问题**：在 Apple Silicon（M 系列芯片）上运行 x86_64 容器时，Kaleido（MultiQC 用于导出静态图表的工具）不兼容。

**日志信息**：
```
WARNING: Detected a Rosetta process, implying that MultiQC is running inside an x86_64 container, but hosted by Apple Silicon.
Static plot export uses Kaleido - a tool that is unstable under conflicting architectures, so MultiQC will not save static PNG/PDF/SVG plots to disk.
```

**解决方案**：无需修复，HTML 报告已包含交互式图表；静态图表可在原生 x86_64 环境中生成。

## 三、nextflow.config 优化

### 3.1 借鉴 riboseq.nf 的改进

#### 3.1.1 离线配置加载修复（高优先级）

**原代码**：
```groovy
includeConfig System.getenv('NXF_OFFLINE') ? 'conf/empty.config' : "${params.custom_config_base}/nfcore_custom.config"
```

**问题**：`conf/empty.config` 文件不存在，导致离线模式下报错。

**改进后**：
```groovy
includeConfig params.custom_config_base && (!System.getenv('NXF_OFFLINE') || !params.custom_config_base.startsWith('http')) ? "${params.custom_config_base}/nfcore_custom.config" : "/dev/null"
```

**改进点**：
- 使用 `/dev/null` 替代不存在的文件
- 支持本地路径配置（即使离线也能加载）
- 添加参数非空检查

#### 3.1.2 iGenomes 配置加载修复（高优先级）

**原代码**：
```groovy
includeConfig params.igenomes_ignore ? 'conf/empty.config' : 'conf/igenomes.config'
```

**改进后**：
```groovy
includeConfig !params.igenomes_ignore ? 'conf/igenomes.config' : 'conf/igenomes_ignored.config'
```

**新增文件**：`conf/igenomes_ignored.config`
```groovy
params.genomes = [:]
```

#### 3.1.3 简化 conda channels 配置

**原代码**：

**改进后**：

```groovy
conda.channels = ['conda-forge', 'bioconda']
```

**改进点**：删除 `defaults` 通道，与 nf-core 标准一致。

#### 3.1.4 添加 arm64 和 emulate_amd64 profile

## 五、SE 数据支持

### 5.1 修改背景

原流程只支持 PE（Paired-End）数据，samplesheet 要求必须有 `fastq_1` 和 `fastq_2` 两列。添加 SE（Single-End）数据支持后，用户可以使用只包含 `fastq_1` 的 samplesheet。

### 5.2 修改内容

#### 5.2.1 主工作流 - `workflows/circdna.nf`

**修改分支逻辑**：
```groovy
.branch {
    meta, fastq ->
        single  : fastq.size() == 1 && meta.single_end
            return [ meta, fastq.flatten() ]
        multiple: fastq.size() > 1 || !meta.single_end
            return [ meta, fastq.flatten() ]
}
```

**修复 skip_trimming 逻辑**：
```groovy
} else {
    ch_trimmed_reads = ch_cat_fastq  // 原代码使用了 INPUT_CHECK.out.reads
}
```

#### 5.2.2 模块支持

以下模块已自动支持 SE 数据（通过 `meta.single_end` 判断）：
- `TRIMGALORE` - 使用 `--single` 模式
- `BWA_MEM` - 自动处理单端比对
- `CIRCLEMAP_READEXTRACTOR` - 处理 BAM 文件，无需区分 SE/PE

### 5.3 支持的 samplesheet 格式

**SE 数据**：
```csv
sample,fastq_1,fastq_2
SAMPLE_SE,/path/to/sample_R1.fastq.gz,
```

**PE 数据**：
```csv
sample,fastq_1,fastq_2
SAMPLE_PE,/path/to/sample_R1.fastq.gz,/path/to/sample_R2.fastq.gz
```

### 8.1 SAMTOOLS_FAIDX 输出未传递到主工作流

**问题描述**：主工作流 `workflows/circdna.nf` 第 224 行尝试使用 `SAMTOOLS_FAIDX.out.fai`，但该模块未在主工作流中导入，导致运行时错误。

- `SAMTOOLS_FAIDX` 在 `BAM_PREPROCESSING` 子工作流中执行，但 `fasta_fai` 输出没有传递回主工作流
- 主工作流尝试直接使用 `SAMTOOLS_FAIDX.out.fai`，但该模块未导入

**修复方案**：
1. 在 `BAM_PREPROCESSING` 子工作流的 `emit` 块中添加 `fasta_fai` 输出
2. 在主工作流中使用 `BAM_PREPROCESSING.out.fasta_fai` 替代 `SAMTOOLS_FAIDX.out.fai`

**修改文件**：
- `subworkflows/local/bam_preprocessing/main.nf` - 添加 `fasta_fai` 输出
- `workflows/circdna.nf` - 使用子工作流的输出

**关键代码**：
```groovy
// bam_preprocessing/main.nf - 添加输出

// circdna.nf - 使用子工作流输出
AMPLICONARCHITECT_PIPELINE (
    ch_bam_sorted,

    BAM_PREPROCESSING.out.fasta_fai,  // 修复前: ch_fasta_meta.join(SAMTOOLS_FAIDX.out.fai)
```

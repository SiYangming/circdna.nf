## 5. CIRCLEMAP_REALIGN 运行时错误修复（方案 1：工作流/配置层）

### 4.1 现象

`-profile test` 运行 `NFCORE_CIRCDNA:CIRCDNA:CIRCLEMAP_REALIGN (circdna_3)` 时报：

```
Missing output file(s) `*.bed` expected by process ...
Command output:
  2026-06-23 11:25:24: An error happenend during execution. Exiting
Command error:
  [E::idx_find_and_load] Could not retrieve index file for 'circdna_3.qname.sorted.bam'
    0%|          | 0/400 [00:00<?, ?it/s]
  0it [00:01, ?it/s]
```

### 4.2 根因分析

1. `[E::idx_find_and_load]` 仅是 pysam 在打开 `-qbam`（query-name sorted BAM）时尝试加载 `.bai` 失败的 warning，**不致命**（query-name 排序的 BAM 本就不能 `samtools index`，官方 tutorial 也不建议为其建索引）。
2. 真正的退出原因是 Circle-Map Realign 在 `Splitting clusters to processors` 后立即抛 `An error happenend during execution`，对应官方 issue tracker 中多线程下 cluster 切分异常的已知问题，常见于**小测试数据 + 多线程**（test profile 下 `process_high` → 4 cpus，candidate cluster 不足以被均匀切分到 4 个 worker）。
3. Circle-Map `Realign` 模块脚本使用 `--threads $task.cpus`，因此 `task.cpus` 决定其内部进程数。

### 4.3 修复（仅改测试 profile 配置，不动模块、不动全局 modules.config）

为保持 [modules/local/circlemap/realign/main.nf](file:///Users/siyangming/nextflow_nf_core/circdna.nf/modules/local/circlemap/realign/main.nf) 纯净，且**不影响生产环境/大样本下的多线程性能**，仅在 [conf/test_local.config](file:///Users/siyangming/nextflow_nf_core/circdna.nf/conf/test_local.config) 的 `process` 块中追加：

```groovy
process {
    // ... 现有 withLabel 资源覆盖 ...

    // Force single-thread for Circle-Map Realign on small test datasets.
    // Multi-process cluster splitting in Circle-Map raises
    //   "An error happenend during execution. Exiting"
    // when candidate cluster count < threads (typical for tiny test BAMs).
    // Production / large samples should NOT need this override.
    withName: 'CIRCLEMAP_REALIGN' {
        cpus = 1
    }
}
```

[conf/modules.config](file:///Users/siyangming/nextflow_nf_core/circdna.nf/conf/modules.config) 中 `withName: 'CIRCLEMAP_REALIGN'` 块**保持原样**（不写 cpus），生产 profile 沿用 `process_high` 默认资源。

效果：
- `-profile test_local` 下 Circle-Map 单进程执行，规避 `Splitting clusters to processors` 引发的退出；
- 真实数据/大样本运行时（不启用 test_local profile）仍是多线程，性能不打折；
- BAI warning 保留但不影响产出。

### 4.4 为什么大样本不需要该 override（背景说明）

- `[E::idx_find_and_load]` 仅是 pysam 打开 query-name BAM 时找 `.bai` 的 stderr warning，**不致命**；query-name 排序的 BAM 本就不能 `samtools index`。
- 真正失败的 `An error happenend during execution. Exiting` 出现在 Circle-Map 的多进程 cluster 切分阶段：candidate cluster 数 < `--threads` 或某 worker 分到空列表时，子进程抛异常。
- 在真实样本中 candidate cluster 数 >> CPU 数，每个 worker 都能拿到非空数据，因此**多线程一直跑得通**。
- 只有 `test_local` 这类极小数据才稳定踩中该缺陷。

使用 `-resume` 重跑 `CIRCLEMAP_REALIGN`：


预期：该进程产出 `*_circularDNA_coordinates.bed` 与 `versions.yml`。
